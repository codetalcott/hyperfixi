║  Demo 1/3 — Claude Desktop  ║
╚═════════════════════════════╝
Scenario: Japanese prompt → multilingual voice/UI command
Target language: JA
Domain: voice

Inputs:
  EN  click submit
  JA  送信 を クリック

Parsed outputs (from MCP parse_multilingual_intent):
  field        │ EN                               │ JA
  ─────────────┼──────────────────────────────────┼─────────────────────────────────
  action       │ click                            │ click
  domain       │ voice                            │ voice
  confidence   │ 1.00                             │ 1.00
  explicit     │ [click patient:submit]           │ [click patient:送信]

Assertions:
  ✓ action is language-neutral (EN=click, JA=click)
  ✓ domain classification agrees (both → voice)
  ✓ role keys match byte-for-byte: [click patient]
  ✓ LSE schema is identical; role values preserve user-authored identifiers ({"patient":"送信"})

MCP tool call (what the IDE sends):
  {
    "tool": "action__parse_multilingual_intent",
    "arguments": {
      "text": "送信 を クリック",
      "language": "ja",
      "domain": "voice"
    }
  }

║  Summary  ║
╚═══════════╝
  3/3 demos produced language-neutral semantic ASTs.

All three demos demonstrate the core architectural claim:
  A prompt in any supported language yields a semantic AST whose
  action, domain, and role keys are byte-for-byte identical to the
  English equivalent. Role values stay in the user's language.
  The behavior layer is language-neutral by construction — not by
  translation.
