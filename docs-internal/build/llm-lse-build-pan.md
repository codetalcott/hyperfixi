# LLM ↔ LSE Integration Build Plan

## Architecture

All core logic lives in `packages/framework` (unit-testable, no MCP dependency). MCP tools/resources/prompts are thin wrappers in `packages/mcp-server`. Everything is schema-driven — when a domain adds a command, the entire pipeline updates automatically.

```
                        ┌─────────────────────────────────┐
                        │        DomainRegistry           │
                        │  (schemas from 7 domains)       │
                        └──────────┬──────────────────────┘
                                   │
             ┌─────────────────────┼─────────────────────┐
             │                     │                     │
    ┌────────▼────────┐  ┌────────▼────────┐  ┌────────▼────────┐
    │ Pillar 1        │  │ Pillar 2        │  │ Pillar 3        │
    │ System Prompts  │  │ Training Data   │  │ Feedback Loop   │
    │                 │  │                 │  │                 │
    │ prompt-generator│  │ schema-synth    │  │ feedback-fmt    │
    │ MCP Resources   │  │ pair-extractor  │  │ confidence-gate │
    │ MCP Prompts     │  │ JSONL export    │  │ pattern-tracker │
    └────────┬────────┘  └────────┬────────┘  └────────┬────────┘
             │                     │                     │
             └─────────────────────┼─────────────────────┘
                                   │
                        ┌──────────▼──────────┐
                        │   MCP Server        │
```

## Phase 1: System Prompt Generation (Pillar 1)

**Goal:** Auto-generate LLM system prompts from `CommandSchema[]`. Expose as MCP Resources.

### 1A: Prompt Generator Core — `packages/framework`

|                |                                                                                                   |
| -------------- | ------------------------------------------------------------------------------------------------- |
| **New files**  | `src/prompts/types.ts`, `src/prompts/prompt-generator.ts`, `src/prompts/prompt-generator.test.ts` |
| **Tests**      | ~30                                                                                               |
| **Depends on** | Nothing (reads existing schemas)                                                                  |

**Key interfaces:**

```typescript
// types.ts
interface PromptGeneratorConfig {
  domain: string;
  description: string;
  schemas: readonly CommandSchema[];
  outputFormat?: 'explicit' | 'json' | 'both'; // what to teach
  examplesPerCommand?: number; // default 2
  maxTokens?: number; // truncation budget
}

interface GeneratedPrompt {
  text: string; // full system prompt markdown
  sections: PromptSection[]; // for partial inclusion
  metadata: {
    domain: string;
    commandCount: number;
    roleCount: number;
    approximateTokens: number;
  };
}
```

**What it generates from each `CommandSchema`:**

```
### fetch — Fetch data from a URL and deliver to a target element
  Required: source (URL, type: expression)
  Optional: style (json|html|text, type: expression), destination (CSS selector)
  Example:  [fetch source:/api/users style:json destination:#user-list]
  JSON:     { "action": "fetch", "roles": { "source": { "type": "expression", "value": "/api/users" }, ... } }
```

Synthetic examples are generated by enumerating valid role combinations from each schema. Value types are sampled from `expectedTypes` with heuristic defaults (`selector` → `#example`, `expression` → `/api/example`, `duration` → `30s`, etc.).

The prompt generator also synthesizes the "meta" sections:

- Protocol overview (LSE syntax summary)
- Value type reference (`selector`, `literal`, `reference`, `flag`, `duration`)
- Per-command documentation (from schemas)
- Output format specification
- Error recovery instructions

## Phase 2: Training Data Generation (Pillar 2)

**Goal:** Generate `(natural_language, LSE)` pairs. Export as JSONL for fine-tuning.

### 2A: Training Data Core — `packages/framework`

|                |                                                                                                                                                 |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| **New files**  | `src/training/types.ts`, `src/training/schema-synthesizer.ts`, `src/training/pair-extractor.ts`, `src/training/jsonl-writer.ts`, + 2 test files |
| **Tests**      | ~45                                                                                                                                             |
| **Depends on** | Phase 1A (`generateExamples`)                                                                                                                   |

**Key types:**

```typescript
interface TrainingPair {
  id: string;
  natural: string; // "fetch /api/users as json into #list"
  language: string; // "en"
  explicit: string; // "[fetch source:/api/users style:json destination:#list]"
  json: SemanticJSON; // LLM-simplified wire format
  domain: string; // "flow"
  action: string; // "fetch"
  source: 'synthetic' | 'test' | 'compilation';
  confidence?: number;
  quality: number; // 0-1, for filtering
}
```

**Three data sources:**

| Source              | Method                                                | Volume                                                   |
| ------------------- | ----------------------------------------------------- | -------------------------------------------------------- |
| Synthetic           | Enumerate role combinations from schemas              | ~5 × 9 commands × 8 languages = 360 pairs for flow alone |
| Test extraction     | Parse existing test inputs through DSL, render as LSE | Thousands (from 8100+ tests)                             |
| Compilation capture | Every `CompileResponse.normalizedSemantic` is a pair  | Unbounded (runtime)                                      |

Schema synthesizer generates examples per command by:

1. Required roles only (minimum viable)
2. Required + each optional role individually
3. Required + all optional roles (maximum)
4. Shuffled role order (teaches role-label independence)

For each combination, uses `markerOverride` per language + `GrammarTransformer` to build natural language forms in all 8 languages.

## Phase 3: Feedback Loop (Pillar 3)

**Goal:** Close the loop — structured error feedback, confidence-aware disambiguation, pattern hit-rate tracking.

### 3A: Structured Error Feedback (Level 1) — `packages/framework`

|                |                                                                                                          |
| -------------- | -------------------------------------------------------------------------------------------------------- |
| **New files**  | `src/feedback/types.ts`, `src/feedback/feedback-formatter.ts`, `src/feedback/feedback-formatter.test.ts` |
| **Tests**      | ~20                                                                                                      |
| **Depends on** | Nothing (uses existing `Diagnostic` types)                                                               |

```typescript
interface LSEFeedback {
  accepted: boolean;
  input: { format: 'explicit' | 'json' | 'natural'; text: string };
  diagnostics: FeedbackDiagnostic[];
  hints: string[]; // actionable retry guidance
  schema?: {
    // valid structure for attempted command
    action: string;
    requiredRoles: string[];
    optionalRoles: string[];
    roleDescriptions: Record<string, string>;
  };
  correctedExample?: {
    // "did you mean this?"
    explicit: string;
    json: SemanticJSON;
  };
}

interface FeedbackDiagnostic {
  severity: 'error' | 'warning' | 'info';
  code: string;
  message: string;
  suggestion?: string;
  fixType?: 'missing_role' | 'invalid_type' | 'unknown_command' | 'unknown_role' | 'syntax_error';
}
```

Maps existing diagnostic codes to machine-actionable `fixType` values. Generates hints like:

- `missing_role` → `"Required role 'source' missing. It expects a URL (type: expression)."`
- `unknown_role` → `"Role 'target' not recognized for 'fetch'. Valid roles: source, style, destination."`
- `unknown_command` → `"Command 'get' not recognized. Did you mean 'fetch'?"`

Generates `correctedExample` by filling in defaults for missing roles from the schema.

### 3B: Confidence-Aware Disambiguation (Level 2) — `packages/framework`

|               |                                                                           |
| ------------- | ------------------------------------------------------------------------- |
| **New files** | `src/feedback/confidence-gate.ts`, `src/feedback/confidence-gate.test.ts` |
| **Tests**     | ~10                                                                       |

```typescript
interface DisambiguationRequest {
  input: string;
  candidates: Array<{
    action: string;
    confidence: number;
    explicit: string; // canonical LSE for this interpretation
  }>;
  question: string; // human/LLM-readable disambiguation prompt
}

function checkDisambiguation(
  confidence: number,
  low?: number, // default 0.5
  high?: number // default 0.7
): boolean;

function buildDisambiguation(
  input: string,
  candidates: Array<{ action: string; confidence: number; node: SemanticNode }>
): DisambiguationRequest;
```

When confidence is in the borderline `0.5–0.7` range, instead of accepting or rejecting, present alternatives:

```
Your input parsed with confidence 0.62.
  (a) [toggle patient:#button]     — toggle class on #button (conf: 0.62)
  (b) [toggle patient:.button]     — toggle .button class on me (conf: 0.58)
Which did you mean?
```

The LLM's answer becomes a training signal.

### 3C: Pattern Hit-Rate Instrumentation (Level 3) — `packages/framework`

|               |                                                                           |
| ------------- | ------------------------------------------------------------------------- |
| **New files** | `src/feedback/pattern-tracker.ts`, `src/feedback/pattern-tracker.test.ts` |
| **Tests**     | ~15                                                                       |

```typescript
interface PatternEvent {
  timestamp: number;
  domain: string;
  action: string;
  language: string;
  inputFormat: 'explicit' | 'json' | 'natural';
  confidence: number;
  outcome: 'accepted' | 'rejected' | 'disambiguated';
  diagnosticCodes?: string[];
}

class PatternTracker {
  record(event: PatternEvent): void;
  getHitRateByCommand(): Record<string, { attempts: number; successes: number; rate: number }>;
  getHitRateByLanguage(): Record<string, { attempts: number; successes: number; rate: number }>;
  getTopFailures(limit?: number): Array<{ code: string; count: number; actions: string[] }>;
  getSummary(): PatternTrackerSummary;
  exportJSONL(): string;
}
```

Ring buffer of last 1000 events. Patterns with high `rejected/attempts` ratios → confidence is miscalibrated for that pattern. Feeds back into adjusting weights or refining patterns.

## Phase 4: Integration and Generalization

**Goal:** Wire everything together. Make opt-in trivial for domains.

### 4A: DomainDescriptor Enhancement

Add `schemas` to `DomainDescriptor` (optional, non-breaking):

```typescript
interface DomainDescriptor {
  // ... existing fields ...
  schemas?: readonly CommandSchema[]; // NEW: enables prompt/training/feedback
}
```

Wire schemas for all 7 domains in `domain-registry-setup.ts`. Each domain already exports its schemas.

### 4B: DomainRegistry Methods

```typescript
class DomainRegistry {
  // ... existing ...
  generatePrompt(domain: string, config?: Partial<PromptGeneratorConfig>): GeneratedPrompt | null;
  generateTrainingData(domain: string, config?: Partial<SynthesisConfig>): SynthesisResult | null;
  buildFeedback(
    domain: string,
    input: string | SemanticJSON,
    diagnostics: Diagnostic[]
  ): LSEFeedback | null;
}
```

### 4C: End-to-End Tests (~20)

- Generate prompt for `flow` → contains all 9 commands
- Synthesize training data → JSONL roundtrips
- Parse synthetic examples → all validate at confidence 1.0
- Invalid explicit syntax → structured feedback with hints
- Borderline confidence → disambiguation request
- Record 100 events → verify hit-rate statistics
- Same pipeline works for `flow`, `sql`, `bdd`, `llm` domains

## Dependency Graph

```
1A (Prompt Generator) ←─── independent
├── 1B (MCP Resources)
├── 1C (MCP Prompts)
└── 2A (Training Data)
     └── 2B (MCP Training Tools)

3A (Error Feedback) ←────── independent, parallelizable with Phase 1
3B (Disambiguation) ←────── independent
3C (Pattern Tracker) ←───── independent
  └── 3D (MCP Feedback Tools)

4 (Integration) ←─────────── depends on all above
```

Phases 1A, 3A, 3B, 3C can run in parallel. Phase 2A depends on 1A. Phase 4 is the final integration pass. Pilot everything on `domain-flow` first (richest schemas, all word orders), then generalize.
