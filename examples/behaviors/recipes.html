Inline Recipes — most "behaviors" are just short scripts

Before reaching for install BehaviorName, check whether a short inline _="…" script does the job. These recipes are adapted from the upstream _hyperscript cookbook (www/patterns/), which is almost entirely inline patterns — not behavior definitions. Promote a recipe to an installed behavior only when it is genuinely reusable and parameterized. They run on the full hyperfixi.js bundle.

Toggle a class

The hello-world of inline scripting. No behavior needed.

_="on click toggle .fun on me"

Reusable + parameterized? Then it's worth promoting — that's exactly install Toggleable(cls).

Fade out and remove

Transition a property, then remove the element.

_="on click transition *opacity to 0 over 300ms then remove me"
Click to dismiss me
…and me

The reusable, parameterized version is install Removable(effect: 'fade').

Character counter

Write to a sibling <output/> on every input — no IDs, no wiring.

_="on input put my value's length into the next <output/>"
0 / 40

For a live reactive version (bind $msg to me), use the hyperfixi-hx-v4.js reactive bundle.

Filter a list by search

The show … when form filters visibility against a condition.

_="on keyup
    show <blockquote/> in the next <div/>
      when its textContent contains my value"

When to promote a recipe to a behavior

A recipe earns install when it is (a) used in several places and (b) parameterized. Outside-click dismissal is a good example — it's a reusable primitive, so it ships as ClickOutside rather than as copy-pasted inline script: 

<div _="install ClickOutside
         on clickoutside add .hidden to me"> … </div>
